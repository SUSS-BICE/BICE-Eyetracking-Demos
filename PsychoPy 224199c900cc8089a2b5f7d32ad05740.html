Setting Up the Eye-Tracker Device

PsychoPy has beef with neon.local for some reason (jk it’s some issue with .local passing through mDNS traffic via the ioHub integration).

  • Go to Experiment Settings > Eyetracking > Companion address > [insert IP address from Companion app]
  • Go to your Command line shell and type ping neon.local
    • Check if there is any packet loss

We’ll go ahead and use pupil_labs.realtime_api.simple to print on the console whether our eye-tracker is connected. Note: You can find the cookbook here

  • Insert this as a custom Code Component > Begin experiment* tab
    • You should place this in your Starting Instructions component
#Discovering device
from pupil_labs.realtime_api.simple import discover_one_device

print("🔍 Attempting to discover Pupil Labs Neon device...")

try:
    device = discover_one_device(max_search_duration_seconds=5)
    if device is None:
        print("❌ No device found on the network. Check your connection or hostname.")
    else:
        print("✅ Connected to device.")
        print(f"🕶️  Glasses Serial Number: {device.serial_number_glasses}")
        #Gaze datum is the coordinates packet being received
        gaze = device.receive_gaze_datum()
        print("📍 Gaze data received.")
except Exception as e:
    print("❌ Error while attempting to connect to device:")
    print(e)

⚠️ The Clock Drift Issue

Whenever we’re using two devices to simultaneously collect data, we have to be aware of the clock drift issue. As a result, we must account for time offset between Pupil Cloud and PsychoPy. We’ll manually calculate it down to the nanosecond before sending it to Pupil Cloud.

  • Insert this as a custom Code Component > Begin experiment* tab
    • You should place this in your Starting Instructions component
#Estimating clock offset
estimate = device.estimate_time_offset()
clock_offset_ns = round(estimate.time_offset_ms.mean * 1_000_000)
# Store device for future use
thisExp.offset_ns = clock_offset_ns

Because we also want to call to the device throughout the experiment, we’re going to go ahead and store the device class as a global variable. I don’t completely understand why. Do it anyway.

# Store device in a global variable instead (not saved)
global pupil_device
pupil_device = device

Manually Shutting Down Connection between PsychoPy & Neon

Sometimes, for reasons unknown, we have to manually shut the connection between PsychoPy and neon in order to prompt the experiment to self-terminate

  • Insert this as a custom Code Component > End Routine* tab
    • You should place this in your Ending Instructions component
try:
    pupil_device.close() #Remember we stored device as a global var, pupil_device
except Exception as e:
    print(f"⚠️ Error closing Pupil Labs device: {e}")
#Get PsychoPy to terminate the experiment
core.quit()

Images

➤ Preserving Aspect Ratio

  • Setting Layout > Size [w,h] to (none, ‘fixed number’) or vice versa will attempt to keep the file’s aspect ratio but I find that it does distort the image. Do the following instead
    • Do note that this is under the assumption that exp settings: Screen > Units > height
    • If there is any height constraints (i.e. can’t use 1.0 units, change accordingly)
    • ⚠️ Only works for local PsychoPy
      • You’ll need to load aspect ratio and image details in the spreadsheet. Refer here
    1. Add a code component above the image component
      from PIL import Image

# Build full path to the image
img_path = poster + '.jpg'  # assuming poster is a string like "poster1"

# Load the image using PIL to get pixel size
im = Image.open(img_path)
img_width_px, img_height_px = im.size
image_aspect_ratio = img_width_px / img_height_px

# Get window aspect ratio from PsychoPy
win_width_px, win_height_px = win.size
win_aspect_ratio = win_width_px / win_height_px

# Max image height in PsychoPy height units
max_image_height = 1.0

# Decide scaling based on aspect ratio
if image_aspect_ratio > win_aspect_ratio:
    # Wider than screen: scale by width
    image_height = max_image_height / image_aspect_ratio
    image_size = [max_image_height, image_height]
else:
    # Taller: scale by height
    image_size = [image_aspect_ratio * max_image_height, max_image_height]

# Optional: print for debug
print(f"Image size set to: {image_size}")
# Optional: print for debug
print("Event name:", poster + '_event')
    1. Go to the Image component. Layout > Size [w,h] > $image_size

Event Annotation

This is specifically for experiments which require an annotation to be triggered. Although Pupil Labs does have a specific plEvent component, the following code accounts for the clock drift issue

  • Insert this as a custom Code Component > Begin Routine* tab
#Annotating event start
label = poster + "_start"
timestamp_ns = time.time_ns() - thisExp.offset_ns
pupil_device.send_event(label, event_timestamp_unix_ns=timestamp_ns)
  • Insert this as a custom Code Component > End Routine* tab
#Annotating event end
timestamp_ns = time.time_ns() - thisExp.offset_ns
pupil_device.send_event(poster + "_end", event_timestamp_unix_ns=timestamp_ns)

Pupil Labs does have a guide on automating event annotation for OpenCV. We’re not using that but you can see the reference here.

Loop

PsychPy requires pretty much everything to be laid out in .xlsx files when it comes to loops.

Loop Type

randompresent a - b in a random order. If we have nReps at 2, this would be repeated twice e.g. [c, a, b, a, c, b]
fullRandompresent a - b in a random order but also take into account the number of nReps i.e. imagine that rather than having 3 items in the bag that we sample from, and repeat this twice, we instead have 6 items int he bag that are randomly sampled from. This would mean that with fullRandom, but not random, it would be possible to get the following order of trials e.g. [a, a, b, c, c, b]
sequential present the rows in the order they are set in the spreadsheet
staircasefor use with adaptive procedures, create an output variable called level that can then be used to set the parameter of a stimulus (e.g. it’s opacity) in an adaptive fashion. This allows researchers to converge upon a participants threshold by adjusting the value of level in accordance with performance.
interleaved staircasefor use with multiple staircases that are interleaved. This can also be used to implement other staircasing algorithms such as QUEST (Watson and Pelli, 1983) via QuestHandler

We’ll talk about staircases another time

For the sake of sanity, we’ll talk about this in the form of Trials and Blocks. As a result, we’ll be discussing a Nested Loop featuring the following:

  • 2 randomised Blocks; each block featuring 5 randomised Trials

Trials

As you can see, we’ve specified each image stimuli to be presented in excel files. There are two files, to represent two blocks

Because we want to randomise both blocks and trials, we’ll leave the block randomisation to our outer loop. For the current inner loop, we’ll specify which excel file should run using $block, which is a reference to the column label ‘block’ that we’ve specified in blockorder.xlsx

  • Conditions > $block + '_poster_trials.xlsx’

Block

This excel file, blockorder.xlsx, specifies the name of our blocks− which also corresponds to group names/labels we’ve used in our trial excel files. Keep your names (and capital/non-capital letters) constant!
